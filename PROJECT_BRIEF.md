# 新项目简报

## 1. 项目背景

当前仓库 `AITradingSimulator` 是一个基于 Flask + SQLite 的多彩种 AI 预测平台，现阶段核心职责是：

- 采集彩种数据
- 生成预测结果
- 保存预测与结算结果
- 提供公开方案展示和收益模拟

它当前并不负责以下能力：

- 使用 Telegram 个人账号自动发言
- 管理 Telegram 会话与群组执行任务
- 聚合第三方预测来源
- 对多来源信号做订阅、融合、风控和执行编排

因此，新需求不应继续在当前仓库内扩展，而应单独新建项目。

## 2. 已形成的关键结论

### 2.1 不建议直接在当前仓库内做自动投注执行

原因：

- 当前项目的运行时扩展维度是“彩种”，不是“执行渠道”。
- PC28 预测存储模型是固定字段模型，暂不适合快速承载更多投注玩法与执行语义。
- 自动投注执行属于高风险外部动作，和预测核心应隔离部署、隔离故障域、隔离权限。
- 后续如果要接更多网站方案、频道方案、公开分享方案，核心问题会变成“信号平台”，而不是“单一 Telegram 执行器”。

### 2.2 不建议把 Telegram-Panel 当核心业务底座

`Telegram-Panel` 更适合被看作一个可选执行端，而不是整个平台的核心。

原因：

- 它解决的是 Telegram 账号执行和后台管理，不是预测聚合平台问题。
- 你们现有主能力与后续规划的核心是：来源接入、标准化、订阅、策略、风控、执行回执。
- 如果未来要接入更多外部预测来源，执行器必须是平台下游，而不是平台中心。
- 它是 .NET 技术栈；当前项目是 Python 技术栈。直接以其为核心会提高整体维护复杂度。

### 2.3 推荐方向

新项目应设计为：

- 平台核心：预测信号聚合与标准化平台
- 执行端：Telegram 自动发言执行器
- `Telegram-Panel`：后续可选适配器或自托管执行端之一

## 3. 目标产品形态

新项目应同时支持两类用户：

### 3.1 托管用户

- 用户不自己部署服务器
- 用户将 Telegram 执行交给平台托管
- 平台负责账号执行环境、任务调度、发送日志和异常处理

### 3.2 自托管用户

- 用户自己部署执行端
- 用户不把 Telegram 账号托管给平台
- 用户通过插件或适配器连接平台，平台只提供预测信号和控制接口

这意味着平台必须把“信号生产”和“信号执行”彻底分离。

## 4. 推荐总体架构

### 4.1 四层模型

#### A. 来源接入层 Source Adapters

负责接入不同预测来源，例如：

- 平台自有 AI 方案
- 其他用户公开分享的方案
- 第三方网站提供的预测计划
- Telegram 频道/群公开内容
- 后续其他格式的数据源

每个来源都应先保留原始数据，再进行解析。

#### B. 信号标准化层 Signal Normalization

将所有来源统一转换为平台内部标准信号结构，而不是直接生成群消息文本。

建议核心字段至少包括：

- `signal_id`
- `source_type`
- `source_id`
- `source_item_id`
- `lottery_type`
- `issue_no`
- `bet_type`
- `bet_value`
- `confidence`
- `published_at`
- `raw_payload`
- `normalized_payload`
- `status`

#### C. 决策与订阅层 Strategy Engine

负责：

- 用户订阅哪些来源
- 多来源冲突如何处理
- 是否允许混合策略
- 金额策略
- 群组映射
- 发送时间控制
- 幂等控制
- 风控限制

#### D. 执行层 Executors

负责将标准信号转换为具体执行动作。

第一阶段可考虑两种执行端：

- 自研 Python Telegram Executor
- `Telegram-Panel` 模块适配器

后续可扩展更多执行端，但执行端只消费标准信号，不应理解复杂来源逻辑。

## 5. 为什么优先建议“自己做平台核心”

如果以后只做单一 Telegram 自动发言，使用现成面板作为基础设施还有讨论空间。

但如果目标包括：

- 接更多第三方预测来源
- 接别人分享的方案
- 做统一订阅平台
- 做托管版和自托管版
- 未来支持多个执行端

那么自己掌握平台核心是必须的。

否则后期会出现以下问题：

- 来源逻辑和执行逻辑耦合
- 迁移执行器成本过高
- 多来源融合越来越难
- 多租户隔离能力不足
- 第三方项目升级和授权会反向限制平台路线

## 6. `Telegram-Panel` 的建议定位

### 6.1 可接受的定位

- 一个可选执行器底座
- 一个自托管用户可安装的执行面板
- 一个内部验证执行流程的 MVP 工具

### 6.2 不建议的定位

- 不建议作为整个平台核心业务底座
- 不建议作为唯一执行方式
- 不建议在没有许可证确认前直接作为商业产品基础

### 6.3 许可证风险

在评估阶段发现，`Telegram-Panel` 仓库当前未看到明确的开源许可证信息。

在没有明确许可证前，不应默认可以：

- 商用
- 修改后分发
- 提供闭源托管服务
- 为外部客户大规模交付

是否采用它，必须先确认作者授权或仓库许可证状态。

## 7. MVP 建议范围

第一阶段应尽量收敛，不要一开始做成大而全平台。

### 7.1 应做

- 平台账户系统
- 方案来源管理
- 标准信号模型
- 用户订阅关系
- 最小风控与幂等
- Telegram 执行器
- 执行日志与回执
- 群组格式模板

### 7.2 暂不做

- 通用插件市场
- 复杂多来源加权投票系统
- 自动资金管理全套策略
- 支持所有 PC28 变体玩法
- 多通道执行
- 面向外部开发者的开放生态

这符合 KISS 和 YAGNI。

## 8. PC28 领域建模注意事项

当前旧仓库中的 PC28 预测结果是一组固定字段，主要围绕：

- 和值
- 大小
- 单双
- 组合

但业务参考玩法里还包括：

- 极大 / 极小
- 对子
- 顺子
- 豹子
- ABC
- 边 / 中 / 大边 / 小边

因此，新项目不要继续沿用“固定几列字段”思路，而应优先考虑：

- 通用投注项模型
- 通用开奖结果派生模型
- 可扩展赔率规则模型
- 可扩展群消息格式映射

否则每加一种玩法都要改表、改接口、改前端、改执行器。

## 9. 平台与执行器的接口原则

执行器不应该理解整个平台内部所有业务细节。

平台对执行器输出的应是“可执行信号”，例如：

- `signal_id`
- `issue_no`
- `bet_type`
- `bet_value`
- `stake_plan`
- `message_text`
- `target_group`
- `execute_after`
- `expire_at`
- `idempotency_key`

执行器回传的应是：

- `signal_id`
- `executor_id`
- `delivery_status`
- `delivered_at`
- `message_id`
- `raw_result`
- `error_message`

## 10. 推荐开发路线

### 阶段一：新建平台骨架

- 新建独立仓库
- 先完成领域模型、数据库设计、接口边界
- 明确“来源”“标准信号”“订阅”“执行任务”四个核心对象

### 阶段二：先做自研 Telegram Executor

原因：

- 技术栈统一
- 最可控
- 不受第三方面板授权和内部结构影响
- 更适合作为托管版基础设施

### 阶段三：再评估 `Telegram-Panel` 适配器

如果许可证和可维护性都没有问题，再做：

- `Telegram-Panel` 插件
- 面向自托管用户的接入方式

## 11. 新项目建议目录方向

仅建议，不要求一次定死：

```text
new-project/
  docs/
  platform/
  executor/
  adapters/
  migrations/
  tests/
```

其中：

- `platform/` 负责来源、标准化、订阅、任务编排
- `executor/` 负责 Telegram 发言执行
- `adapters/` 负责第三方来源与第三方执行端适配

## 12. 下一步建议

新仓库建立后，优先做以下工作：

1. 确认项目名、技术栈和部署方式
2. 设计数据库核心表
3. 设计标准信号结构
4. 明确托管版与自托管版的边界
5. 先做最小闭环：
   - 接入一个来源
   - 产出标准信号
   - 执行到一个 Telegram 群
   - 回写执行结果

- 新项目目标是“预测信号平台 + Telegram 执行体系”
- 平台核心优先自研
- `Telegram-Panel` 仅作为后续可选执行适配
