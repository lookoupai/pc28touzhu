# 架构说明

## 1. 设计目标

本项目要解决的问题不是“单次自动发消息”，而是“多来源预测信号到 Telegram 执行”的完整链路。

系统必须支持：

- 平台自有预测方案
- 外部网站预测方案
- 公开分享方案
- Telegram 频道/群来源
- 后续更多来源与更多执行端

因此，核心设计原则是：

- 来源与执行解耦
- 先标准化，再执行
- 平台与执行器独立部署
- 托管版与自托管版共用同一套信号协议

## 2. 核心分层

### 2.1 来源接入层

负责采集原始数据，不直接做执行。

职责：

- 连接外部来源
- 拉取或接收原始方案内容
- 保留原始载荷
- 生成待标准化记录

### 2.2 标准化层

负责把不同来源转为统一信号。

职责：

- 统一 `lottery_type`
- 统一 `issue_no`
- 统一 `bet_type`
- 统一 `bet_value`
- 统一置信度、发布时间、来源标识

### 2.3 策略层

负责从“标准信号”得到“可执行任务”。

职责：

- 用户订阅关系
- 来源过滤
- 冲突处理
- 群组映射
- 金额策略
- 幂等控制
- 到期控制

### 2.4 执行层

执行器只关心两件事：

- 收到什么任务
- 如何把任务发出去并回执

执行器不负责来源解析，不负责策略融合，不负责平台业务判断。

## 3. 部署形态

### 3.1 托管版

- 平台与执行器都由服务方部署
- 用户将账号执行交给平台
- 适合没有服务器的用户

### 3.2 自托管版

- 平台由服务方提供
- 执行器由用户自己部署
- 适合不愿托管 Telegram 账号的用户

## 4. `Telegram-Panel` 的位置

`Telegram-Panel` 不作为平台核心，只作为可选执行器适配方案。

允许的使用方式：

- 作为一个自托管执行器底座
- 作为后续兼容某类用户环境的适配目标

不建议的使用方式：

- 让整个平台绑定在它的内部结构上
- 让平台核心业务依赖其数据模型和权限模型

## 5. 第一阶段落地边界

第一阶段只关心最小闭环，不追求大而全：

- 单一来源接入
- 单一 PC28 信号标准化
- 单一 Telegram 执行器
- 单一回执链路

以下内容明确不在第一阶段内：

- 多来源投票融合
- 通用插件市场
- 多执行通道
- 全量 PC28 玩法覆盖
- 面向第三方开发者的开放平台
