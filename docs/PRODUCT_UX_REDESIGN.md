# 产品与体验改造方案

## 1. 文档目的

本文档用于统一 `pc28touzhu` 的产品定位、信息架构与第一阶段前端改造范围。

目标不是单纯“把现有页面做漂亮”，而是把当前偏工程对象视角的控制台，逐步调整为：

- 面向托管用户的自动投注工作台
- 面向平台运营/开发的管理控制台

## 2. 当前判断

### 2.1 真实产品定位

项目的真实目标不是纯后台工具，而是：

- 第一正式来源接入 `AITradingSimulator`
- 平台托管用户的 Telegram 执行账号
- 自动将预测结果发到指定群组执行
- 同时兼容托管版与自托管版

这意味着产品对外形态是用户可直接使用的 C 端托管产品，但底层仍然保持“信号平台 + 执行器”解耦。

### 2.2 当前页面问题

当前根路径页面把底层对象直接暴露给用户：

- `source`
- `raw item`
- `signal`
- `subscription`
- `delivery target`
- `execution job`
- `executor`
- `alert`

这更像平台内部运维面板，而不是普通托管用户的使用入口。

## 3. 产品分层

建议拆成两层界面：

### 3.1 用户端

面向托管用户，围绕“开通和使用自动投注”。

核心目标：

- 导入想跟随的预测方案
- 绑定托管 Telegram 账号
- 绑定投注群组
- 配置跟单策略
- 查看执行结果与异常

### 3.2 管理端

面向平台运营、调试与高级管理，保留当前对象控制能力。

核心目标：

- 管理来源
- 检查 raw item / signal / dispatch
- 检查执行器与告警
- 排查失败任务

## 4. 信息架构

### 4.1 用户端导航

第一阶段建议保留一个轻量入口页，围绕以下模块展开：

- 首页
- 我的托管
- 自动投注
- 执行记录
- 异常提醒

第一阶段为了控制实现成本，可先把这些能力浓缩在首页中展示，使用跳转入口衔接现有管理端。

当前已落地：

- `/` 用户端首页
- `/records` 用户端执行记录页

### 4.2 管理端导航

当前控制台下沉为 `/admin`，保留现有对象管理能力：

- 来源管理
- Telegram 账号
- 订阅管理
- 投递目标
- 标准信号
- 原始内容
- 执行任务
- 执行器
- 告警
- 最近失败任务

## 5. 用户主流程

### 5.1 托管用户最小主线

1. 注册或登录
2. 导入 `AITradingSimulator` 方案
3. 绑定托管 Telegram 账号
4. 配置投递目标群组
5. 建立跟单关系
6. 观察执行结果与异常

### 5.2 第一阶段方案导入方式

第一阶段推荐主流程不是“手写 JSON 配置”，而是：

1. 用户从 `AITradingSimulator` 复制方案导出链接
2. 在 `pc28touzhu` 管理端导入该链接
3. 平台创建本地来源并进入后续抓取/标准化/派发流程

后续可演进为：

- 接入码导入
- 分享链接导入
- 账号打通后直接读取方案列表

## 6. 第一阶段界面策略

### 6.1 根路径 `/`

新根路径作为用户端首页，不再直接显示底层对象表格。

首页职责：

- 说明产品主线
- 展示当前账号状态
- 展示当前配置完成度
- 展示最近执行情况与异常
- 提供去管理端完成配置的明确入口

首页只回答用户最关心的问题：

- 我是否已登录
- 是否已经具备自动执行条件
- 当前有几个方案/账号/群组/订阅
- 最近执行成功还是失败
- 有没有需要立刻处理的异常

当前已落地的第一阶段补充能力：

- 用户端首页可直接导入 `AITradingSimulator` 公开方案页链接或导出链接
- 导入动作底层仍复用现有 `source` 模型
- 高级来源配置仍保留在 `/admin`
- 用户端已拆出独立的执行记录页与异常提醒页
- 用户端自动投注配置页已支持最小启停控制
- 用户端自动投注配置页已支持全局暂停/恢复聚合开关
- 用户端已新增独立的自动投注配置页
- 首页会根据当前缺失项动态给出“下一步”主 CTA

### 6.2 管理端 `/admin`

现有控制台整体保留，但文案与入口定位改为“管理控制台”。

管理端不再充当产品首页，只服务：

- 平台配置
- 高级调试
- 数据校验

## 7. 页面职责映射

### 7.1 用户端首页展示的数据

基于现有接口可直接复用：

- `GET /api/auth/me`
- `GET /api/platform/sources`
- `GET /api/platform/telegram-accounts`
- `GET /api/platform/subscriptions`
- `GET /api/platform/delivery-targets`
- `GET /api/platform/execution-jobs`
- `GET /api/platform/alerts`

首页不直接修改底层对象，只做：

- 状态聚合
- 进度提示
- 最近执行结果展示
- 前往管理端的导航

### 7.2 管理端继续承担的能力

继续使用现有接口与对象：

- 来源新增与抓取
- raw item 标准化
- signal 派发
- 任务重试
- 执行器监控
- 告警与失败任务查看

## 8. 视觉方向

### 8.1 用户端

关键词：

- 可信
- 清晰
- 低学习成本
- 高风险动作可感知

设计要求：

- 以状态卡、步骤卡、最近记录为核心
- 减少大表格与大段 JSON
- 明确展示“下一步该做什么”
- 对错误与风险用强对比提示

### 8.2 管理端

关键词：

- 工具化
- 信息密度高
- 排障友好

设计要求：

- 保留表格与对象明细
- 保持管理员高效率操作
- 通过文案明确其不是普通用户首页

## 9. 第一阶段代码改造范围

本轮只做最小可落地改造，不引入新的复杂后端模型。

### 9.1 应做

- 新增用户端首页 `/`
- 支持用户端直接粘贴 `AITradingSimulator` 公开方案页链接或导出地址导入来源
- 现有控制台迁移到 `/admin`
- 新增用户端静态资源
- 为管理端增加锚点与入口链接
- 更新测试，覆盖 `/`、`/admin` 与新静态资源
- 更新 README 中的入口说明

### 9.2 暂不做

- 新的分享码/接入码协议
- AITradingSimulator 账号打通
- 用户端单独的策略配置新接口
- 用户端真正的一键启停自动投注开关
- 全新的管理后台路由体系

## 10. 后续迭代建议

### 阶段二

- 做“导入预测方案”专用流程
- 隐藏用户端对 `source_id`、`telegram_account_id` 等内部字段的感知
- 将用户端首页拆分为“自动投注 / 执行记录 / 异常提醒 / 我的托管”

### 阶段三

- 支持分享链接或接入码导入
- 支持直接选择 `AITradingSimulator` 方案
- 补齐策略页与托管状态页
- 对接更明确的用户状态 API

## 11. 方案结论

第一阶段不应继续把现有对象控制台当作产品首页，而应：

- 把产品首页改成托管用户工作台
- 把现有页面定位为管理控制台
- 在不破坏现有执行链路的前提下，先完成角色分层

这是当前最符合 KISS、YAGNI 且可直接落地的改造方式。
